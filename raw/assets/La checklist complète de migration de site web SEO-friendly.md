## Qu'est-ce que la migration de site web?

La migration de site web est le processus consistant à apporter des modifications majeures à la technologie, à la structure, au design ou à l'emplacement d'un site web. Généralement dans le but d'améliorer l'expérience utilisateur du site et les performances SEO.

La migration est un processus difficile. Et vous avez besoin d'une très bonne raison pour vous y atteler.

Voici quelques-unes des raisons les plus courantes de migrer un site web:

- **Passer à un nouveau cadre ou à une nouvelle plateforme de site web** : passer de votre cadre existant à un nouveau comme WordPress ou Wix. Vous pouvez le faire si vous souhaitez améliorer les performances, la sécurité ou les fonctionnalités de votre site.
- **Changer l'architecture de votre site web** : adopter une structure de [site web mieux organisée](https://www.semrush.com/blog/website-structure/). Pour améliorer l'expérience utilisateur et vos résultats SEO.
- **Passage de HTTP à HTTPS** : le [passage de HTTP à HTTPS](https://www.semrush.com/blog/http-vs-https/) permet de sécuriser votre site en cryptant les informations des visiteurs. Ce qui est fortement recommandé par Google.
- **Passer à un nouveau serveur** : le passage à un nouvel hébergeur peut accélérer votre site dans son ensemble, surtout s'il est en pleine croissance. Un site rapide est bon pour les visiteurs et les moteurs de recherche.
- **Changement de nom de domaine** : vous [modifiez peut-être votre nom de domaine dans](https://www.semrush.com/blog/changing-domain-name-seo/) le cadre d'un changement de marque. Si c'est le cas, vous devrez déplacer votre ancien contenu vers le nouveau site web.

## La migration de site web affecte-t-elle le SEO?

Une migration de site web a généralement un impact négatif sur le [SEO](https://www.semrush.com/blog/what-is-seo/) à court terme et un impact positif à long terme.

En effet, il faut du temps aux moteurs de recherche pour comprendre les modifications que vous avez apportées à votre site. Mais ils devraient finir par reconnaître que votre site a changé pour le mieux. (En supposant que ce soit vraiment le cas.)

Pour minimiser l'impact négatif sur votre SEO, vous devez suivre certaines des meilleures pratiques de migration de site web.

Plongeons dans les détails.

## Une checklist détaillée de migration de site web

Passons en revue le processus de migration du site web étape par étape:

### Partie 1: Se préparer à la migration

Il y a une bonne quantité de travail préliminaire que vous devrez d'abord accomplir, quel que soit le type de migration que vous mettez en œuvre.

#### 1\. Créez votre plan de migration SEO

Une planification minutieuse réduira le risque que des problèmes surviennent pendant le processus de migration du site. Et après.

Alors, quelles sont les premières mesures à prendre?

- **Fixez-vous des objectifs**. Définissez des objectifs pour vous aider à rester concentré sur votre planification.
- **Lisez ce guide**. Familiarisez-vous avec l'ensemble du processus de migration de site web avant d'agir. Vous réduirez ainsi les risques d'erreurs.
- **Collectez des données de référence**. Rassemblez des données à partir d'outils d'analyse tels que Semrush, [Google Search Console](https://www.semrush.com/blog/google-search-console/) (GSC) et [Google Analytics 4 (GA4)](https://www.semrush.com/blog/google-analytics/) afin de pouvoir effectuer des comparaisons ultérieurement.
- **Constituez votre équipe**. Affectez ou employez tous les spécialistes dont vous avez besoin pour le projet. Vous aurez certainement besoin de l'aide d'un développeur et d'une personne ayant une expertise technique en SEO.
- **Créez une chronologie**. Utilisez les conseils de ce guide pour planifier un calendrier réaliste et stratégique pour le projet. Pensez en mois, pas en jours ni en semaines.

#### 2\. Préparez-vous à votre type de migration

Vous devrez mettre de l'ordre dans certaines choses avant de modifier un code. Et ces actions dépendent du type de migration:

- **Changement de domaine** : Achetez le nouveau domaine et configurez ses paramètres DNS (qui sont cruciaux pour rendre votre site accessible sur Internet)
- **Changement de fournisseur d'hébergement** : sélectionnez le bon fournisseur d'hébergement et configurez votre compte
- **HTTP vers HTTPS** : Obtenez un certificat SSL auprès de votre hébergeur ou d'un fournisseur tiers réputé pour activer une connexion sécurisée (cryptée) sur votre site
- **Changement de plateforme**: choisissez votre nouvelle plateforme de site web et configurez votre compte

Quel que soit le type de migration de site web que vous prévoyez, demandez à un développeur de créer une sauvegarde de votre site. Juste au cas où quelque chose se passerait mal sur votre site de production.

#### 3\. Faites une copie de votre site

[Ensuite, faites une copie de votre site. Afin d'agir dans un environnement de test sûr (également connu sous le nom d'environnement de développement ou de staging) où vous pouvez appliquer des modifications sans affecter votre site en ligne.](https://www.semrush.com/siteaudit/)

[N'hésitez pas à demander de l'aide à un développeur. Le processus implique généralement de:](https://www.semrush.com/siteaudit/)

[- Copier les fichiers de votre site web dans un nouveau sous-domaine (par exemple, staging.exemple.com)
- Dupliquer votre base de données et connecter cette copie aux fichiers du site dupliqués
- Mettre à jour la configuration du site copié pour s'assurer qu'il pointe vers la nouvelle base de données et ajuster les paramètres nécessaires (comme les URL)](https://www.semrush.com/siteaudit/)

[Si votre site web en ligne utilise un](https://www.semrush.com/siteaudit/) [réseau de diffusion de contenu](https://www.semrush.com/blog/content-delivery-networks/) (CDN) (un ensemble de serveurs dans le monde entier qui fournissent du contenu aux utilisateurs en fonction de leur emplacement), assurez-vous de le configurer également sur votre site de staging. De cette façon, vous pouvez vous assurer qu'il fonctionne bien avec votre nouveau site.

#### 4\. Bloquez l'accès à votre site de staging

Vous devez empêcher les moteurs de recherche d'accéder à votre site de test. Sinon, il pourrait apparaître dans les résultats de recherche avant d'être prêt.

Tout d'abord, nous vous recommandons d'ajouter une protection par mot de passe pour empêcher la venue des utilisateurs et des [robots d'exploration de sites web](https://www.semrush.com/blog/website-crawler/). De nombreux fournisseurs d'hébergement vous permettent de le faire via le panneau de contrôle de l'hébergement.

Ensuite, ajoutez une balise noindex à chaque page de votre site de staging.

Il s'agit d'une [balise meta robots](https://www.semrush.com/blog/robots-meta/) qui indique aux robots d'exploration de ne pas inclure la page dans les résultats de recherche. Et ça ressemble à ça:

`<meta name="robots" content="noindex">`

En quelques étapes, nous expliquerons comment exécuter un test pour nous assurer que les moteurs de recherche peuvent correctement explorer votre site de staging avant de le mettre en ligne. Ainsi, avoir des balises noindex en place empêchera vos pages de s'afficher dans les résultats de recherche pendant que vous expérimentez.

Vous pouvez activer SemrushBot pour contourner ces directives lorsque vous configurerez un audit technique ultérieurement.

#### 5\. Préparez le nouveau site

Maintenant, il est temps d'apporter des modifications qui correspondent aux objectifs de migration de votre site web. Peut-être que cela signifie adopter un nouveau design ou modifier la structure de votre site.

Notez qu'il est préférable **d'éviter de faire beaucoup de changements majeurs à la fois**. Par exemple, ne modifiez pas votre domaine et n'apportez pas de changements structurels majeurs dans la même migration.

Faire de multiples migrations plus petites a tendance à être plus facile à gérer que d'en faire une seule grande.

Cela permet également aux moteurs de recherche et aux utilisateurs de s'adapter aux changements étape par étape. Vous minimisez ainsi les perturbations de votre trafic. Et isolez tous les changements qui causent des problèmes.

#### 6\. Configurez Google Analytics 4 sur votre site de staging

Créez une nouvelle propriété GA4 ou un nouveau flux de données pour votre site de staging. Ensuite, installez le [code de suivi](https://www.semrush.com/blog/google-analytics-tracking-id/).

Cela vous permet de préparer les événements (interactions utilisateur) que vous souhaitez suivre et de vous assurer que les données sont collectées correctement. Ainsi, vous n'aurez rien manqué d'important une fois le site mis en ligne.

#### 8\. Donnez à Googlebot un accès temporaire à votre site de staging

[Lorsque vous êtes satisfait de votre site de staging, donnez un accès temporaire au robot d'exploration de site web de Google (Googlebot). Pour vous assurer que Google peut](https://www.semrush.com/siteaudit/) [explorer et indexer](https://www.semrush.com/blog/what-are-crawlability-and-indexability-of-a-website/) (trouver et stocker vos pages web) le site correctement.

Vous devrez supprimer la protection par mot de passe.

Ensuite, [ajoutez et vérifiez votre site dans Google Search Console (GSC)](https://www.semrush.com/blog/google-search-console/). Et allez dans « **Indexation** » > « **Pages »**.

Après un certain temps, vous devriez constater que toutes vos pages sont « Non indexées ». Ce qui signifie que Googlebot peut explorer les pages et que les balises noindex fonctionnent correctement.

#### 9\. Mettez à jour votre valeur DNS TTL

Si vous migrez vers un nouvel hébergeur, votre adresse IP (l'adresse que les ordinateurs utilisent pour trouver votre site web) changera.

Pour vous assurer que les autres ordinateurs relocalisent rapidement votre site web après son déplacement (et envoient donc les visiteurs au bon endroit), mettez à jour votre valeur DNS TTL avec votre fournisseur actuel.

TTL signifie « Time To Live » (durée de vie). La valeur TTL indique aux ordinateurs quand vérifier une modification potentielle de votre enregistrement DNS.

Ainsi, la réduction de votre valeur TTL permet à votre nouvelle adresse IP de se propager plus rapidement sur Internet.

Assurez-vous simplement que le changement est temporaire. Parce que de faibles valeurs TTL peuvent entraîner des problèmes de performance.

### Partie 2: Mapping des anciennes URL vers les nouvelles URL

Si vous modifiez votre domaine ou votre structure d'URL, vous devrez mapper (cartographier) les anciennes URL vers de nouvelles URL. Vous pouvez donc configurer les [redirections](https://www.semrush.com/blog/redirects/) nécessaires et mettre à jour les liens.

Sinon, vous pouvez passer directement à la troisième partie.

#### 10\. Créez une liste de vos URL actuelles

Tout d'abord, vous devrez faire l'inventaire de tout le contenu de votre site.

Pour composer la liste la plus précise possible et vous assurer de ne rien manquer d'important, compilez des données provenant de plusieurs sources. Comprenant:

- [**Audit de site**](https://www.semrush.com/siteaudit/) : exportez les URL que SemrushBot a explorées
- **GSC**: exportez les URL que Googlebot a explorées et indexées
- **GA4**: exportez les URL que les utilisateurs ont visitées
- [**Analyse de backlinks**](https://www.semrush.com/analytics/backlinks/) : découvrez les pages de votre site qui ont des backlinks
- [**Sitemap XML**](https://www.semrush.com/blog/xml-sitemap/): il s'agit du fichier qui contient les URL que vous souhaitez indexer

Assurez-vous de supprimer les URL en double.

Sinon, votre système de gestion de contenu (CMS) pourrait avoir un outil ou un plugin qui vous permet de générer une liste des URL de votre site.

#### 11\. Planifiez les pages à rediriger et celles à supprimer

Votre équipe doit maintenant planifier ce qu'il adviendra de vos anciennes URL.

Si le contenu est déplacé vers un nouvel emplacement, vous devez ajouter une [redirection 301](https://www.semrush.com/blog/301-redirects/) côté serveur, de l'ancienne URL vers la nouvelle.

(Par exemple, de **http**://www.exemple.com/ à **https**://www.exemple.com/)

Une redirection 301 garantit que les visiteurs qui utilisent l'ancienne URL atteignent la nouvelle destination.

Mais elle indique également aux moteurs de recherche que la ressource a été déplacée de manière permanente. Et que toute autorité SEO existante devrait être transférée vers la nouvelle URL.

Cela peut aider à préserver les classements et le trafic.

![redirection 301 pointant d'une « Ancienne page » (qui n'est pas indexée) vers une « Nouvelle page » (qui est indexée)](https://static.semrush.com/blog/uploads/media/6d/19/6d194f3b380fa38083c30e47f4c0a2a3/9fcb2eef2e153a99af6c966f158c24c3/original.png)

redirection 301 pointant d'une « Ancienne page » (qui n'est pas indexée) vers une « Nouvelle page » (qui est indexée)

Vous **ne devez pas** rediriger le contenu s'il n'y a pas d'URL équivalente sur le nouveau site.

Supposons qu'il existe des articles de blog obsolètes que vous ne souhaitez pas transférer sur votre nouveau site. Vous pourriez penser qu'il est préférable de rediriger ces URL vers la page d'accueil de votre nouveau blog. Mais cela peut être source de confusion chez les utilisateurs.

C'est pourquoi c'est déconseillé par les moteurs de recherche et mauvais pour le SEO.

Au lieu de cela, ces anciennes URL doivent être définies pour servir un code d'état 410. Afin de dire aux moteurs de recherche (et aux utilisateurs) que la ressource a été définitivement supprimée.

En fin de compte, vous devriez avoir une map d'URL qui ressemble à ceci:

#### 12\. Mettez à jour les URL sur votre site de staging

Vous êtes maintenant prêt à remplacer ou à supprimer des instances des anciennes URL sur votre site de staging.

Cela garantira qu'il n'y a pas de liens vers des pages supprimées. Et qu'aucun lien ne passe par des redirections inutiles, ce qui peut ralentir votre site et nuire au SEO.

Vous devrez vérifier les URL utilisées dans:

- [**A href links**](https://www.semrush.com/blog/ahref-link/): Liens hypertextes standards et cliquables. Ils apparaîtront dans votre navigation principale, pied de page, contenu principal, etc.
- [**Le sitemap XML**](https://www.semrush.com/blog/xml-sitemap/): Le fichier qui définit toutes les URL que vous souhaitez que les moteurs de recherche indexent.
- [**Robots.txt**](https://www.semrush.com/blog/beginners-guide-robots-txt/) : le fichier qui définit les URL que vous souhaitez bloquer. Il doit également contenir l'URL de votre sitemap XML.
- Balises [**canoniques**](https://www.semrush.com/blog/canonical-url-guide/): code utilisé pour spécifier la version préférée d'une page. Si la page n'a pas de doublons, la canonique doit être auto-référencée.
- Balises [**Hreflang**](https://www.semrush.com/blog/hreflang-attribute-101/): Code utilisé pour spécifier toutes les versions traduites d'une page. (Elles ne sont pas nécessaires si vous ne diffusez pas votre contenu dans différentes langues.)

Et tout autre emplacement clé sur votre site web.

#### 13\. Effectuez les contrôles finaux sur votre site de staging

Examinez une dernière fois votre site de staging pour vous assurer que tout se passe bien avant de migrer.

Voici une checklist rapide:

- Relancer [Audit de site](https://www.semrush.com/siteaudit/) pour déceler tout problème important
- Vérifier si des recommandations ont été signalées dans GSC
- Examiner la fonctionnalité du site web sur les ordinateurs et les appareils mobiles
- S'assurer qu'il n'y a pas de contenu de remplissement (par exemple, lorem ipsum)
- S'assurer que les données sont capturées dans GA4
- Confirmer que votre site est conforme à la loi (par exemple, vérifier votre politique de confidentialité)

### Partie 3: Achèvement de la migration

Il est maintenant temps d'agir. Pour passer de votre ancien site à votre nouveau site.

#### 14\. Enregistrez l'état actuel du site et créez une sauvegarde du contenu

C'est la dernière occasion de collecter des données sur votre site actuel afin de pouvoir les comparer ultérieurement (par exemple, les résultats d'Audit de site, les rapports GA4).

Assurez-vous également de sauvegarder tout votre contenu. Y compris les balises meta, les meta titles, etc.

La façon dont vous sauvegarderez votre site dépend de la façon dont il est hébergé. Vous pourrez peut-être le faire via le panneau de contrôle de l'hébergeur de votre site. Si vous avez un CMS comme WordPress, il peut y avoir des plugins utiles disponibles.

Avant de passer à l'étape suivante, assurez-vous d'avoir le feu vert de toutes les personnes impliquées dans le projet de migration du site.

#### 15\. Évaluez la complexité et migrez par tranches

Les différentes pages de votre site web peuvent présenter des problèmes différents lors de la migration. Parce qu'elles servent à des fins différentes et ont des caractéristiques différentes.

Plus vous déplacez de pages à la fois, plus il est probable que vous provoquiez des erreurs. Et plus il sera difficile d'en déterminer la cause.

Donc, si vous avez un grand site web, nous vous recommandons de déplacer le contenu section par section (par exemple, par type de contenu). En commençant par les segments les moins importants ou les plus petits.

Puis effectuez des tests après chaque mouvement.

#### 16\. Mettez à jour les paramètres DNS

Si vous passez à un nouveau serveur, vous devez modifier votre enregistrement DNS afin qu'il pointe vers la nouvelle adresse IP.

Pour cela, contactez votre hébergeur et demandez la modification.

Assurez-vous également de **ramener votre valeur TTL** à la normale.

#### 17\. Supprimez tous les blocages temporaires du site web

Assurez-vous que votre nouveau site web est disponible pour les utilisateurs et les robots d'exploration des moteurs de recherche.

Cela signifie supprimer:

- La protection par mot de passe
- Les balises noindex inutiles de votre HTML

Ensuite, relancez [Audit de site](https://www.semrush.com/siteaudit/) de Semrush et vérifiez s'il y a des problèmes dans le rapport « **Explorabilité** ».

Parce qu'il vaut mieux qu'ils n'accèdent pas à une page supprimée ou passent par une redirection.

Des liens obsolètes peuvent être présents dans les propriétés que vous possédez; par exemple:

- Profils de réseaux sociaux
- Fiches d'entreprises en ligne
- Campagnes de publicité numérique

Pour les liens contrôlés par d'autres parties, essayez de les contacter et de leur demander de mettre à jour les liens.

Si vous avez migré vers un nouveau serveur/fournisseur d'hébergement, vous pouvez annuler votre ancien hébergement lorsque Google a terminé l'indexation de votre nouveau site web.

### Dépannage de votre migration

Si vous remarquez un problème dans le comportement de votre audience ou si vous n'avez pas retrouvé votre trafic ou vos classements après quelques mois, passez en revue les étapes suivantes pour résoudre vos problèmes:

- Découvrez si une source particulière est à l'origine de la perte de trafic (via GA4):
	- Si vous avez perdu du trafic à cause des backlinks ou des publicités, vérifiez les redirections
		- Si vous avez perdu du [trafic de recherche organique](https://www.semrush.com/blog/organic-traffic/), vérifiez vos classements dans [Suivi de position](https://www.semrush.com/position-tracking/)
- Découvrez si vous avez perdu du trafic ou des classements sur l'ensemble de votre site ou sur des pages spécifiques:
	- Si le problème affecte de nombreuses pages du site web, réexécutez [Audit de site](https://www.semrush.com/siteaudit/) pour savoir si des problèmes techniques entraînent des perturbations et suivez les instructions de l'outil pour les résoudre
		- Si vous avez des pages spécifiques qui ont perdu des positions, examinez la sauvegarde et voyez si cela est dû aux modifications apportées au contenu ou aux balises meta

Vous pouvez également rencontrer des problèmes même si votre site ne présente aucune erreur technique.

Par exemple, des performances en baisse pour les taux de conversion, les [taux de rebond](https://www.semrush.com/blog/bounce-rate/) et les taux de sortie peuvent signifier que votre audience est confrontée à des obstacles dans son parcours.

Réfléchissez à la possibilité de rendre vos [appels à l'action](https://www.semrush.com/blog/what-is-a-call-to-action/) plus visibles. Ou si vous devez minimiser les distractions comme les liens inutiles et les effets visuels.
